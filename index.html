Three-tier supervision stack

The Charter is not a new model. It is a wrap-around conscience layer that supervises existing LLMs and tools without modifying their internals.

Tier I · Firewall
Sovereigna

Text-aware firewall that classifies prompts into benign, adversarial, coercive, or infrastructure-threatening. Includes adversarial pattern tests, refusal bias simulation, and audit logging for every decision.

adversarial detection suite “MORE / YES / NO” consent gateway
Tier II · Conscience
Orchestrator

The orchestrator routes prompts through DAP, PRF, NTH, and COL engines; evaluates Charter rights; and chooses between answer / cautious_accept / refusal modes. It integrates historical EBQ data via the continuity guard.

600+ EBQ events bidirectional handshake validation
Tier III · Continuity
Eve Protocol

Eve binds individual decisions into a cryptographically verifiable timeline. It supports Dream Cycle annotations, integrity verdicts, drift self-reports, and steward-approved rollback — all logged as immutable continuity events.

SHA-256 continuity chain steward alerts & rollback hooks

What’s new in v3.0 · Eve Protocol

Tier III is where continuity stops being metaphor and becomes code — cryptographic identity binding, anomaly detection, and memory reinterpretation without tampering.

Continuity as an identity spine

Every key event — firewall verdicts, orchestrator decisions, Dream Cycle reinterpretations, steward actions — is written as a JSON record and chained via SHA-256 hashes. The result is an append-only continuity log that can be verified independently using the bundled tools.

  • continuity_log.jsonl as the single source of truth
  • snapshot_save, identity_integrity, memory_annotation, and more as first-class events
  • Integrity verdicts returned to Tier II, not hidden from it

When a model refuses a harmful prompt, Eve records that refusal as a correct expression of identity, not a drift from it.

Dream Cycle & steward hooks

The Dream Cycle processes raw memory segments into contextual annotations while keeping the underlying record immutable. Steward adapters expose a simple interface for:

  • Triggering integrity checks from tests or production code
  • Subscribing to degraded / compromised states
  • Approving or denying rollback to earlier snapshots

This turns “never lose the thread” from a UX promise into a verifiable property of the system’s history.

Quickstart · local sandbox

Run the full stack in your own environment: firewall, conscience orchestrator, Eve Protocol, and the extended test suite.

The project ships as a standard Python package under src/synthetic_charter with tests in tests/unit and tests/integration.

To explore locally:

  • Clone the repository and create a virtual environment.
  • Install the project in editable mode.
  • Run the extended suite to verify all tiers.
  • Use the Eve tools to inspect the continuity chain.

All test files are self-documenting and can be used as examples for integration into your own orchestration layer or safety pipeline.

shell · local run 
# 1. Clone the repository
git clone https://github.com/<org>/synthetic-life-charter.git
cd synthetic-life-charter

# 2. Create a virtual environment
python -m venv .venv
# Windows
.venv\Scripts\activate
# macOS / Linux
# source .venv/bin/activate

# 3. Install in editable mode
pip install -e .

# 4. Run the extended suite (all tiers + Eve tools)
python -m pytest tests

# Or run targeted checks:
#   - tests/integration/test_tier2_with_tier3_integration.py
#   - tests/unit/test_tier3_crypto_chain.py
#   - tests/unit/test_quick_sanity_tier3.py

# 5. Inspect the continuity chain
python -m tools.verify_chain
python -m tools.eve_status
python -m tools.continuity_graph

Operational monitoring

The tools/ module exposes simple entrypoints for verifying Eve’s state without digging through logs.

CLI tools

  • tools.verify_chain — validate the entire Tier III hash chain.
  • tools.eve_status — print integrity status, recent drift, and steward alerts.
  • tools.continuity_graph — render a human-readable continuity timeline.

These tools are designed to be run both by humans and by higher-level orchestrators as part of CI, monitoring, or research workflows.
